**What happened**: BM25 always returns results, even when none are relevant. The word "en" appeared in math formulas and the word "pizza" appeared in unrelated contexts (example sentences, vocabulary drills, social engineering anecdotes). The top score (7.34) looks confident, but the result is completely wrong.

#### Vector Search — Semantic Similarity (kNN)

| # | Filename | Score |
|---|----------|-------|
| 1 | Business Vocabulary in Use - Advanced | 0.5929 |
| 2 | Thinking, Fast and Slow (Kahneman) | 0.5929 |
| 3 | How to Solve It (Pólya) | 0.5929 |
| 4 | Machine Learning with PyTorch (Raschka) | 0.5580 |
| 5 | Making Numbers Count (Chip Heath) | 0.5487 |

**What happened**: The top three results share the exact same score (0.5929) — these are books where the PDF parser extracted zero words, so the embedding model encoded an empty string and produced identical vectors. None of the results are relevant, but the scores tell us that clearly:

- **Relevant queries** in the tests above scored **0.70–0.84**
- **This irrelevant query** scored only **0.55–0.59**

The low scores are the system saying "I am not confident about any of these results."

#### Why This Matters for RAG Systems

This test reveals a critical difference between keyword search and vector search when the query has no good match:

| Behaviour | BM25 (keyword) | Vector (semantic) |
|-----------|---------------|-------------------|
| Returns results? | Yes, always | Yes, always |
| Scores look confident? | **Yes** — top score was 7.34 | **No** — top score was only 0.59 |
| Can you set a threshold? | Hard — scores vary wildly between queries | **Yes** — scores are normalised (0 to 1) |
| Risk of misleading the user? | **High** — confident but wrong | **Low** — low score signals uncertainty |

In a production RAG system, you would set a **score threshold** (for example, 0.65) on the vector search results. If no document passes the threshold, the system responds with "I could not find relevant information" instead of sending irrelevant context to the LLM. This is the first line of defence against **hallucination** — the retrieval layer must be able to say "I don't know."

BM25 scores are harder to threshold because they are not normalised — a score of 7.0 might be excellent for one query and terrible for another.

| Scenario | Best system | Why |
|----------|-------------|-----|
| Filter by file type, size, date | **MongoDB** | Structured metadata queries are instant |
| Find exact phrases or technical terms | **Elasticsearch** | BM25 excels at precise keyword matching |
| Find conceptually similar content | **Vector search** | Embeddings capture meaning, not just words |
| User asks a question (not keywords) | **Vector search** | Questions rarely share exact words with answers |
| Combine all three | **Hybrid** | Filter → keyword rank → semantic re-rank |

## How to Improve This Pipeline

This pipeline works, but it has four clear weaknesses that a production RAG system would need to fix. Each one is a learning opportunity — the reading links explain the concepts behind each improvement.

---

### 1. Chunking — Don't Embed the Whole Document at Once

**The problem**: We currently encode only the first 512 characters of each document. A 300-page book has its embedding generated from the first paragraph alone. The remaining 99% of the content is invisible to semantic search.

**The solution**: Split each document into overlapping chunks (500–1000 tokens each) and store each chunk as a separate vector. When someone searches "How to build an AI app", the system can find the specific chapter or paragraph that answers the question — not just the book that might contain the answer somewhere.

**Key design decisions**:
- **Chunk size**: Too small (50 tokens) loses context. Too large (5000 tokens) dilutes the signal. 500–1000 tokens is the common sweet spot.
- **Overlap**: Chunks should overlap by 10–20% so that sentences split across chunk boundaries are still searchable.
- **Metadata**: Each chunk should carry a reference back to its parent document (filename, page number, chapter) so you can cite the source.

**Reading**:
- [Chunking Strategies for LLM Applications (Pinecone)](https://www.pinecone.io/learn/chunking-strategies/) — visual guide to different chunking approaches and their trade-offs
- [LangChain Text Splitters](https://python.langchain.com/docs/how_to/#text-splitters) — practical implementations of recursive, token-based, and semantic chunking
- [Evaluating Chunking Strategies for RAG (Unstructured.io)](https://unstructured.io/blog/chunking-for-rag-best-practices) — benchmarks showing how chunk size affects retrieval quality
- [Greg Kamradt: 5 Levels of Text Splitting (YouTube)](https://www.youtube.com/watch?v=8OJC21T2SL4) — character → recursive → semantic → agentic splitting explained in 20 minutes

---

### 2. Hybrid Search — Combine Keyword and Vector Results

**The problem**: We run three separate searches and display them side by side. In production, you want a single ranked list that benefits from both keyword precision and semantic understanding.

**The solution**: **Reciprocal Rank Fusion (RRF)** merges two ranked lists into one. If a document ranks high in both BM25 and vector search, it gets boosted to the top. If it ranks high in only one, it still appears but lower. Elasticsearch 8.x supports this natively.

**How RRF works**:
```
RRF_score = Σ  1 / (k + rank_in_list)
```
For each document, sum its reciprocal rank across all lists. The constant `k` (typically 60) prevents top-ranked documents from dominating too aggressively.

**Reading**:
- [Reciprocal Rank Fusion (Elasticsearch docs)](https://www.elastic.co/guide/en/elasticsearch/reference/current/rrf.html) — official guide with query examples
- [Hybrid Search Explained (Qdrant)](https://qdrant.tech/articles/hybrid-search/) — concept explainer with diagrams showing how keyword + vector fusion works
- [Hybrid Search with Elasticsearch (Elastic blog)](https://www.elastic.co/blog/improving-information-retrieval-elastic-stack-hybrid) — step-by-step implementation using ES kNN + BM25 together
- [RAG Fusion: Going Beyond Naive RAG](https://arxiv.org/abs/2402.03367) — research paper on generating multiple query variations and fusing their results

---

### 3. Score Thresholding — Let the System Say "I Don't Know"

**The problem**: The pizza query test showed that our system always returns 5 results, even when none are relevant. If this pipeline fed into an LLM, the model would try to generate an answer from irrelevant documents — this is how hallucination starts.

**The solution**: Set a minimum similarity score on vector search results. Since cosine similarity scores are normalised between 0 and 1, you can set a meaningful threshold:
- **Above 0.70**: Likely relevant — send to the LLM
- **0.55–0.70**: Uncertain — maybe include with a warning
- **Below 0.55**: Not relevant — respond with "No relevant documents found"

**Why this is hard with BM25**: BM25 scores are not normalised. A score of 7.0 might be the best result for one query and a terrible result for another. This is why vector search is more reliable for threshold-based filtering.

**Reading**:
- [Relevance Thresholds in Vector Search (Weaviate)](https://weaviate.io/blog/distance-metrics-in-vector-search) — how cosine, dot product, and L2 distances relate to relevance
- [Reducing Hallucination in RAG (LlamaIndex)](https://docs.llamaindex.ai/en/stable/optimizing/production_rag/) — production RAG guide covering thresholds, re-ranking, and fallback strategies
- [When RAG Goes Wrong (Hamel Husain)](https://hamel.dev/blog/posts/rag/) — real-world failure modes and how retrieval quality directly controls hallucination rates
- [RAGAS: Evaluating RAG Pipelines](https://docs.ragas.io/en/latest/) — framework for measuring retrieval faithfulness, answer relevancy, and context precision

---

### 4. Stronger Embedding Models — Better Meaning, Higher Cost

**The problem**: We use `paraphrase-multilingual-MiniLM-L12-v2`, which produces 384-dimensional vectors. It is fast and multilingual, but larger models capture meaning more precisely — especially for technical and domain-specific content.

**The solution**: Upgrade to a larger embedding model. The trade-off is always speed vs quality:

| Model | Dimensions | Speed | Quality | Notes |
|-------|-----------|-------|---------|-------|
| `paraphrase-multilingual-MiniLM-L12-v2` | 384 | Fast | Good | What we use now |
| `intfloat/multilingual-e5-large` | 1024 | Medium | Better | Open-source, strong multilingual |
| `BAAI/bge-m3` | 1024 | Medium | Better | Supports dense + sparse + multi-vector |
| `text-embedding-3-small` (OpenAI) | 1536 | API call | Very good | Requires API key and costs per token |
| `text-embedding-3-large` (OpenAI) | 3072 | API call | Best | Most expensive, highest quality |

**Important**: You cannot mix embeddings from different models. If you switch models, you must re-embed all documents. This is why the model choice is an early architectural decision.

**Reading**:
- [MTEB Leaderboard (Hugging Face)](https://huggingface.co/spaces/mteb/leaderboard) — benchmark comparing 100+ embedding models on retrieval, classification, and clustering tasks
- [Choosing an Embedding Model for RAG (Pinecone)](https://www.pinecone.io/learn/series/rag/embedding-models-rag/) — practical guide to model selection with benchmark comparisons
- [BGE-M3: Multi-Functionality Embedding Model](https://huggingface.co/BAAI/bge-m3) — model card for the state-of-the-art open-source option
- [OpenAI Embeddings Guide](https://platform.openai.com/docs/guides/embeddings) — API documentation for the commercial alternative

**What we learned from this test**:

The v2 results are more interesting than v1's, but they expose a nuance about thresholding at the chunk level:

1. **Hybrid RRF did well** — it found paragraphs that genuinely mention pizza (robot ordering pizza, emoji encoding examples, pizza topping chatbot exercises). These are real matches for the word "pizza" in context.

2. **Semantic search scores are surprisingly high (0.76–0.81)** — even though the query is irrelevant to the library's subject matter. Why? Because v2 chunks are short (500 words), and a short paragraph about pizza topping selection is genuinely semantically close to "best pizza recipe" even though it is not actually a recipe.

3. **The threshold (0.65) did not filter these out** — at the chunk level, short focused paragraphs naturally produce higher cosine similarity scores than whole-document embeddings. A threshold that works for document-level search (0.65) is too low for chunk-level search.

**Lesson for production**: Score thresholds must be **re-calibrated** when you change the indexing granularity. Moving from documents to chunks typically requires raising the threshold by 0.10–0.15 (e.g., from 0.65 to 0.80) because chunks are more focused and produce higher similarity scores across the board.

**Key takeaways**:

1. **Chunking is the single biggest improvement**. Scores improved by +0.06 to +0.10 across all queries. More importantly, the system now finds the **specific section** that answers the question, not just the book.

2. **Hybrid search (RRF) combines the strengths of both systems**. The BM25 component catches exact keyword matches while the kNN component catches semantic near-misses. Together they produce more balanced rankings.

3. **Score thresholding needs calibration per granularity level**. A threshold of 0.65 works for whole-document embeddings but is too permissive for chunk-level embeddings. Production systems need to run threshold calibration experiments on a validation set.

4. **The irrelevant query test got harder, not easier**. v1 returned obviously garbage results (empty documents). v2 returns plausible-looking but off-topic results (paragraphs that mention pizza but are not recipes). This is actually a more dangerous failure mode for RAG — the retrieval looks reasonable but misleads the LLM. Solving this requires either a **re-ranker** (a cross-encoder model that scores query-document pairs more carefully) or **query classification** (detecting that the query is out of domain before searching).
